\section{Evaluation}\label{sec:evaluation}

The goal of this evaluation is to assess the applicability of runtimes based on Interaction Nets to real-world problems, identify bottlenecks and issues, and highlight directions for further investigation.

\subsection{Experimental Setup}

\paragraph{Hardware.}
All experiments were conducted on a system with an AMD Ryzen 7 5800X 8-core processor (16 threads with hyper-threading enabled, base clock 3.8 GHz) and 128 GiB of DDR5 RAM.

\paragraph{Software.}
We compare five implementations presented below.
\begin{itemize}
    \item \textbf{F\#}: Our quadtree-based single-threaded implementation, compiled for the .NET 10.0 runtime.
    This serves as a baseline representing a conventional functional language approach and prototype for porting to Interaction Nets.
    BenchmarkDotNet\footnote{\url{https://github.com/dotnet/BenchmarkDotNet}}\cite{!!!} for benchmarking automation (e.g. fro JIT warmup).
	\item \textbf{Inpla}: Our interaction-net-based implementation in Inpla low-level language, executed on the Inpla runtime system (version 0.13.4, patched).
	Our patches of Inpla allow one to use arbitrary heap size configuration with per-thread adjustments via scripting.
    The patch also increases several virtual machine compile-time constants to support very large trees (thus, large graphs).
    Additionally, a bottleneck in lists parsing was discovered during testing, and the fix has since been merged into the main Inpla repository\footnote{\url{https://github.com/inpla/inpla/pull/4}}.
    In this evaluation we use 120 GB of heap.
    Three or five runs depending on data size.
    \item \textbf{Vine}: Our interaction-net-based implementation in Vine high-level language, executed on the Vine runtime system\footnote{!!!}. We set 110 GB of heap per worker. Five runs for each configuration
    \item \textbf{LAGraph}: A collection of graph algorithms built on SuiteSparse:GraphBLAS (version !!!), providing state-of-the-art, linear-algebra-based reference implementations of BFS, SSSP, and triangle counting. We use the provided algorithms as a SOTA baseline for high-performance parallel graph analysis. Also we use LAGraph as a baseline to analyze scaling across available threads.
    Number of runs !!! 
    \item \textbf{NetworkX}: A popular Python library for graph analysis (version !!!).
    We use the algorithms implemented in NetworkX as a well-known baseline.
    We do not use any special backends for NetworkX or special settings.
    Number of runs !!!
\end{itemize}

\paragraph{Dataset.}
We evaluate our implementations on several real-world and synthetic graphs from the SuiteSparse Matrix Collection~\cite{Kolodziej2019}.
The graphs are chosen to represent a variety of sizes, sparsity patterns, and application domains, taking into account the capabilities and limitations of the selected competitors.
Table~\ref{tab:datasets} summarizes number of vertices (\(|V|\)) and number of edges (\(|E|\)) of selected graphs.
Weights were corrected by shifting all weights with respect to minimal weight over particular graph to prevent negative loops, and converted to \texttt{Int} due to Inpla does not support floats.

\paragraph{Research Questions.}
Our evaluation addresses the following research questions:

\begin{description}
    \item[RQ1 (Correctness):] Do the F\# and Inpla implementations produce correct results for BFS and triangle counting on all datasets, when compared against reference outputs from NetworkX and LAGraph?

    \item[RQ2 (Performance -- Inpla vs. Vine vs. State-of-the-Art):] How does the execution time of Inpla compare to LAGraph on BFS, SSSP, and triangle counting?
         We also include F\# and NetworkX as additional baselines representing different points in the performance--productivity trade-off.

    \item[RQ3 (Parallel Scaling):] How well do the Inpla, Vine, and LAGraph implementations scale with the number of available CPU cores?
          We measure speedup from 1 to all available cores for BFS, SSSP, and triangle counting, with hyper-threading both enabled and disabled.

    \item[RQ4 (Impact of Matrix Reordering):] Matrix reordering techniques (e.g., reverse Cuthill--McKee) can improve locality and cache performance.
    \item[] Even for ~\cite{10.1145/3731599.3767441}
          How does applying such reordering affect the performance of Inpla, F\#, and LAGraph implementations?
\end{description}
